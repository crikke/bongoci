# ci2 Design Spec

**Date:** 2026-05-18  
**Status:** Approved

## Overview

`ci2` is a Go CLI that runs build tasks defined in `build.toml` using BuildKit as the execution engine. Tasks run inside a container image, with the project source mounted read-only. All writes go to a scratch `/out` volume. Artifacts (binaries or OCI tarballs) are exported back to the host `./out` directory via BuildKit's local exporter.

This is a rewrite of the C# `ci` project, using BuildKit's native Go client library instead of manual protobuf construction.

## CLI

```
ci run <task> [<task>...]
```

- Searches upward from cwd for `build.toml`
- BuildKit daemon address from `BUILDKIT_HOST` env var (default: `unix:///run/buildkit/buildkitd.sock`)
- Runs tasks sequentially; each is a separate BuildKit solve (BuildKit caching deduplicates shared upstream nodes)

## build.toml DSL

```toml
version = 1

[module]
name = "payments-service"
image = "mcr.microsoft.com/dotnet/sdk:10.0"
dependencies = ["../other-module"]   # sibling dirs, mounted at their abs path

[tasks.restore]
cmd = "dotnet restore --packages /packages"
# no inputs — reads only from context

[tasks.compile]
cmd = "dotnet publish -c Release -o /out"
inputs = [
    { task = "restore", path = "/packages", dest = "/packages" }
]

[tasks.build-image]
type = "docker"
dockerfile = "Dockerfile"
inputs = [
    { task = "compile", path = "/out", dest = "/out" }
]

[outputs.binary]
dest = "./out/payments-service"
from = { task = "compile", path = "/out/payments-service" }
```

- `inputs[].path` — path inside the upstream task's output (the `/out` scratch)
- `inputs[].dest` — where to mount it in the current task's container
- `outputs` — defines what gets copied from BuildKit back to the host

## Package Layout

```
cmd/ci/main.go       — CLI entrypoint
pkg/manifest/        — TOML parse → domain model
pkg/compiler/        — domain model → llb.State DAG
pkg/runner/          — BuildKit client, solve, status streaming, export
```

## Domain Model (`pkg/manifest`)

```go
type Manifest struct {
    Name         string
    Image        string
    Dependencies []string        // relative paths resolved to absolute at parse time
    Tasks        map[string]*Task
    Outputs      map[string]Output
}

type Task struct {
    ID         string
    Cmd        string
    Type       string            // "exec" (default) | "docker"
    Dockerfile string
    Inputs     []TaskInput
}

type TaskInput struct {
    Task *Task
    Path string   // selector into upstream task's output
    Dest string   // mount point in this container
}

type Output struct {
    TaskName string
    SrcPath  string   // path inside task's /out
    DestPath string   // host destination, e.g. "./out/payments-service"
}
```

Parser does two passes: create all tasks first, then wire inputs (to handle forward references). Validates: missing `module.name`/`module.image`, unknown task references in inputs/outputs, cycles via DFS.

## LLB Compiler (`pkg/compiler`)

Compiles a `Manifest` + target task name into an `llb.State`.

**For each `ExecTask`** (topo-sorted, deepest dependency first):

1. Start from `llb.Image(manifest.Image)`
2. Mount `llb.Local("context")` read-only at the module's absolute path
3. Mount one `llb.Local("<dep-name>")` per dependency, read-only at its absolute path
4. Mount each `TaskInput` — a selector into the upstream task's compiled state — at `Dest`, read-only
5. Mount `llb.Scratch()` at `/out` as the single writable output (`WithCustomName` for status display)
6. Call `.Run(llb.Shlex(task.Cmd), mounts...)` — returns a new state whose root is the `/out` scratch

**For `DockerTask`**: use the `dockerfile.v0` BuildKit frontend. Pass compiled input states as named gateway contexts.

**Topo-sort**: DFS from target task, visiting inputs before the task itself. Detects cycles (validated earlier in manifest parse, but compiler asserts too).

## Runner (`pkg/runner`)

BuildKit's local exporter writes the entire `/out` scratch to a directory. Because `Output.SrcPath` may select a sub-path (e.g. `/out/payments-service`), the runner exports to a temp dir, then copies `tempDir/<SrcPath>` to `output.DestPath` on the host. OCI tar tasks export directly to the declared dest path.

```go
c, err := client.New(ctx, host)

solveOpt := client.SolveOpt{
    Exports: []client.ExportEntry{
        // ExecTask → export /out to temp dir, then copy SrcPath → DestPath
        {Type: client.ExporterLocal, OutputDir: tmpDir},
        // DockerTask → OCI tar written directly to DestPath
        {Type: client.ExporterOCI, Output: func(_ map[string]string) (io.WriteCloser, error) { ... }},
    },
    LocalDirs: map[string]string{
        "context":    manifest.AbsPath,
        "<dep-name>": depAbsPath,
        // one entry per dependency
    },
}

ch := make(chan *client.SolveStatus)
go printStatus(ch)   // stream vertex start/done/error

_, err = c.Build(ctx, solveOpt, "", buildFunc, ch)
```

`buildFunc` receives a `gateway.Client`, calls `llbState.Marshal(ctx)` to get the definition, submits via `gwClient.Solve()`, and returns the result ref.

Status lines: `=> <vertex-name>` on start, `DONE <vertex-name>` on complete, `ERROR: <msg>` on failure.

## Error Handling

- `build.toml` not found → print hint, exit 1
- Manifest parse/validation errors → print specific error, exit 1
- Unknown task name in `ci run` → print available tasks, exit 1
- BuildKit connection failure → print hint to check `BUILDKIT_HOST`, exit 1
- Task failure → print vertex error from status stream, exit 1

## Out of Scope

- Parallel multi-task solves (sequential is fine; BuildKit cache handles dedup)
- Image push to registry (OCI tar to `./out` only)
- Non-local contexts (all source is local)
